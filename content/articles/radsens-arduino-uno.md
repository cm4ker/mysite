## Часть 1. Подключаем RadSens

### Шаг 1. Что понадобится

- Arduino UNO (R3 или совместимая).
- RadSens (любая ревизия с I2C).
- Четыре провода «мама-папа» или «папа-папа» — зависит от того, есть ли на RadSens штыри или гнёзда.
- USB-кабель для прошивки.

![Компоненты на столе](/images/radsens-parts.jpg)

### Шаг 2. Распиновка платы

На плате четыре контакта, подписаны прямо на текстолите:

| Пин RadSens | Назначение     | Куда на Arduino UNO |
|-------------|----------------|---------------------|
| VCC         | Питание 3–5 В  | 5V                  |
| GND         | Земля          | GND                 |
| SDA         | Данные I2C     | A4                  |
| SCL         | Clock I2C      | A5                  |

На Arduino UNO `SDA` и `SCL` живут на аналоговых пинах **A4** и **A5**; у новых ревизий они продублированы отдельной парой рядом с разъёмом AREF — можно использовать любые.

![Распиновка платы крупным планом](/images/radsens-pinout.jpg)

### Шаг 3. Сборка схемы

Порядок подключения значения не имеет, но удобно начать с питания и земли, потом подкинуть SDA и SCL. Подтягивающие резисторы на линиях I2C уже стоят на самой плате RadSens — внешние не нужны (почему вообще нужны pull-up — см. [Электрика](#электрика-open-drain-и-pull-up)).

![Готовая схема на столе](/images/radsens-wiring.jpg)

Проверь дважды: перепутанные VCC и GND — самый частый способ убить плату.

### Шаг 4. Установка библиотеки

В Arduino IDE открой **Sketch → Include Library → Manage Libraries**, в поиске набери `RadSens` и поставь библиотеку **CG_RadSens** от ClimateGuard. Она же лежит на GitHub: [climateguard/RadSens](https://github.com/climateguard/RadSens).

![Окно Library Manager с RadSens](/images/radsens-library.jpg)

### Шаг 5. Минимальный скетч

Раз в секунду выводим уровень радиации в мкР/ч и количество импульсов:

```cpp
#include <Wire.h>
#include <CG_RadSens.h>

CG_RadSens radSens;

void setup() {
  Serial.begin(9600);
  Wire.begin();

  if (!radSens.init()) {
    Serial.println("RadSens init failed. Check wiring.");
    while (true) {}
  }

  Serial.print("Firmware: ");
  Serial.println(radSens.getFirmwareVersion());
  Serial.print("Sensor address: 0x");
  Serial.println(radSens.getSensorChipId(), HEX);
}

void loop() {
  Serial.print("Intensity dynamic: ");
  Serial.print(radSens.getRadIntensyDynamic());
  Serial.print(" uR/h | static: ");
  Serial.print(radSens.getRadIntensyStatic());
  Serial.print(" uR/h | pulses: ");
  Serial.println(radSens.getNumberOfPulses());

  delay(1000);
}
```

Заливаем (**Ctrl+U**), открываем **Serial Monitor** на 9600 бод.

### Шаг 6. Что должно быть в Serial Monitor

В норме фон в квартире — 10–25 мкР/ч. Динамическое значение усредняется по короткому окну и шевелится заметнее, статическое — по более длинному, выглядит спокойнее.

![Скриншот Serial Monitor с показаниями](/images/radsens-serial.jpg)

Если в выводе нули или `init failed` — проверяй по этому списку:

- SDA/SCL не перепутаны местами.
- Контакты пропаяны, провода в гнёздах сидят плотно.
- На питании есть честные 5 В (мерь мультиметром между VCC и GND датчика).
- Адрес на шине не занят другим устройством — по умолчанию RadSens живёт на `0x66`. Подробнее в разделе [Адресация и I2C-scanner](#адресация-и-i2c-scanner).

## Часть 2. Как работает I2C

В первой части всё было простым: воткнул четыре провода, залил скетч, получил числа. Теперь — почему хватает всего двух сигнальных линий, чтобы общаться с несколькими устройствами одновременно, и что физически происходит на проводах между Arduino и RadSens.

### Сигнальные линии: SDA и SCL

I2C — это шина с двумя сигнальными линиями:

- **SDA** (Serial Data) — линия данных.
- **SCL** (Serial Clock) — линия тактового сигнала.

На Arduino UNO они выведены на пины **A4 (SDA)** и **A5 (SCL)**. Питание датчика — 3–5 В, поэтому 5V с Arduino подходит напрямую, уровни логики тоже совместимы.

### Master и slave

I2C — это шина с ролями. На каждой передаче есть один **master** — он инициирует обмен и генерирует тактовый сигнал на SCL. Остальные устройства на шине — **slave**, они отвечают только тогда, когда master адресовался именно к ним по их 7-битному адресу.

В нашей связке:

- **Master** — Arduino UNO. Решает, *когда* спрашивать и *что* спрашивать.
- **Slave** — RadSens. Сидит на адресе `0x66`, отдаёт данные по запросу.

На одной шине может висеть несколько slave-устройств одновременно — у каждого свой адрес. Мастеров тоже может быть несколько (multi-master), но в обычной самоделке с Arduino это редкость — почти всегда один master и один-несколько slave.

### Топология шины

В общем виде шина I2C — это **две сигнальные линии** (SDA и SCL) с **одной парой подтягивающих резисторов** (pull-up на VCC), к которым параллельно подцеплены все участники: один мастер и любое разумное количество slave. Каждый slave висит на двух линиях одновременно: и SDA, и SCL приходят к нему.

<inline-svg src="/images/i2c-bus.svg" />

Что важно увидеть на схеме:

- **Pull-up по одному на линию, на всю шину.** Не на каждое устройство — это сразу даст лишнюю просадку и порченные фронты. У большинства модулей с алишки pull-up уже распаян, и если набрать на шину три таких — суммарное сопротивление окажется слишком мало (≈1.5 кΩ вместо 4.7), фронт SCL пойдёт пологий. Лечится отпайкой лишних резисторов на slave-платах.
- **Адреса уникальны.** Если на шину захочется повесить второй RadSens — никак, он жёстко на `0x66`. У BMP280 и SSD1306 есть выбор адреса ножкой, у RadSens — нет.
- **Общая земля.** На картинке не нарисована, но она обязательна: VCC и GND всех устройств должны быть связаны с землёй мастера, иначе уровни логики «поплывут» и шина не заработает.

Длина шины — практическое ограничение в пару десятков сантиметров перемычками; до метра-двух при коротких проводах и низкой скорости. Дальше — повышение помех, заваленные фронты, нужно либо снижать резисторы pull-up, либо ставить I2C-buffer/extender.

### Электрика: open-drain и pull-up

I2C построена на схеме **open-drain**: у каждого устройства на SDA и SCL вместо обычного двухтактного выхода стоит транзистор-«ключ» к земле. Он умеет только **притопить** линию в GND. Поднять её в HIGH ни один из участников шины не может — это делает общий **подтягивающий резистор** (pull-up) на линии, обычно 4.7 кΩ к +V.

<inline-svg src="/images/i2c-opendrain.svg" />

Получается логика **wired-AND**: все устройства «отпустили» линию (транзисторы закрыты) → pull-up подтягивает SDA в HIGH. Любое устройство замкнуло свой ключ → линия прижата к GND, на SDA читается LOW. Достаточно одного.

Бонус схемы: на одну шину можно безопасно прицепить **несколько устройств**, никакой контенции не будет. Если двое «прижмут» SDA одновременно — будет просто LOW, никто не сгорит. На обычном двухтактном (push-pull) выходе один драйверил бы HIGH, второй LOW — это короткое замыкание через выходные транзисторы. Open-drain эту проблему убирает по конструкции.

### Передача байта: тайминг и ACK

Два сигнала, синхронные по битовым слотам. Сверху — часы (clock), снизу — данные для байта `0xA5` (двоично `10100101`). Диаграмма начинается с момента сразу после условия START (см. [следующий раздел](#обрамление-start-и-stop)): SDA уже опущена в LOW мастером, дальше восемь тактов клока переносят биты. На девятом такте — **ACK** от slave: тот притапливает SDA в LOW, подтверждая приём. Заодно это естественно готовит линию к условию STOP: SDA уже внизу, остаётся отпустить её в HIGH при HIGH на SCL. Пунктирные направляющие сквозные: каждый столбец — один такт = один бит.

<inline-svg src="/images/i2c-timing.svg" />

- **SCL** — мастер (Arduino) дёргает её ровными тактами, по два уровня на бит: первая половина LOW, вторая половина HIGH. В простое линия отдыхает в HIGH.
- **SDA** — линия данных. Бит выставляется мастером, пока SCL внизу, и читается приёмником на фронте SCL вверх. Биты идут со старшего к младшему (MSB → LSB) и каждый удерживается на весь свой тактовый слот.

Сам **ACK** — прямое применение wired-AND из предыдущего раздела. В девятом такте мастер «отпускает» SDA (его выходной транзистор закрыт). Если slave замыкает свой транзистор — линия LOW для всех, в том числе для мастера = подтверждение приёма. Если slave ничего не сделал, pull-up держит линию в HIGH = NACK («не подтвердил, ошибка»).

### Обрамление: START и STOP

Чтобы байты данных не «терялись» в общем шуме, каждая транзакция I2C обрамлена двумя специальными состояниями, которые легко отличить от обычной передачи бита: **START** и **STOP**. Их фишка в том, что SDA меняется тогда, когда SCL **в HIGH** — в обычной передаче так делать нельзя, поэтому slave-устройства однозначно распознают эти переходы как служебные.

- **START** (обозначают `S`) — мастер тянет SDA из HIGH в LOW, пока SCL высоко. Сигнализирует: «начинаю обмен, дальше адрес».
- **STOP** (обозначают `P`) — мастер отпускает SDA из LOW в HIGH, пока SCL высоко. Сигнализирует: «обмен закончен, шина свободна».

<inline-svg src="/images/i2c-startstop.svg" />

Между ними укладывается всё содержательное: 7 бит адреса + бит направления R/W (читаем мы из устройства или пишем в него), ACK от slave, дальше байты данных, каждый со своим ACK. На обычный одиночный регистр у RadSens получается: `S` → адрес `0x66` + `W` → регистр → `Sr` (повторный START) → адрес + `R` → байт значения → `P`.

### Адресация и I2C-scanner

7-битный адрес — это «зашитый» в чип идентификатор, который master указывает в начале каждой передачи. Узнать его можно тремя способами:

1. **Документация.** В даташите датчика прямо указано, на каком адресе он отвечает. У RadSens это `0x66`, у популярного BMP280 — `0x76` или `0x77` (выбирается ножкой SDO), у дисплеев SSD1306 — `0x3C` или `0x3D`. Большинство модулей с алишки идут с распаянной заводской ножкой выбора адреса.
2. **Надпись на плате.** Многие модули прямо рядом с контактами печатают что-то вроде `ADDR=0x68`. Заглядывать туда быстрее, чем искать pdf.
3. **I2C-scanner.** Если ничего не известно — заливаем универсальный скетч, который опрашивает все 127 адресов и печатает те, что откликнулись. Он же спасает, когда «датчик не работает»: если в выводе пусто — проблема в проводах или питании, если адрес виден, но другой, чем ожидался — нашёл живое устройство, разбираемся дальше.

Вот минимальный сканер:

```cpp
#include <Wire.h>

void setup() {
  Serial.begin(9600);
  Wire.begin();
  Serial.println("Scanning I2C...");

  byte found = 0;
  for (byte addr = 1; addr < 127; addr++) {
    Wire.beginTransmission(addr);
    if (Wire.endTransmission() == 0) {
      Serial.print("  found device at 0x");
      if (addr < 16) Serial.print("0");
      Serial.println(addr, HEX);
      found++;
    }
  }

  Serial.print("Done. Devices: ");
  Serial.println(found);
}

void loop() {}
```

Принцип простой: на каждый адрес master шлёт «пустой» START → ADDR → STOP. Если slave с таким адресом есть на шине — он подтверждает (ACK), `endTransmission()` возвращает 0. Если нет — таймаут, возвращается ненулевой код.

Заливаем, открываем Serial Monitor — увидим список адресов. Для нашей сборки там должно быть `0x66`.

## Часть 3. Что дальше
